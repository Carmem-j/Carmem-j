Data Navigation

Papel

Designer de Produto

Ano

2023

Duração

3 meses

Equipe

Equipe de Visualização de Dados

1. Visão Geral

O usuário, principalmente de O&G, necessita ter a capacidade de consumir, no mesmo dashboard, informações gerais sobre a vida útil de um poço e eventos menores de interesse dentro dessa visão "macro", representada por uma grande janela temporal. Para proporcionar melhorias de navegação de dados dinâmicos, o Data Navigation foi projetado para que o usuário possa interagir de forma mais fluida e intuitiva nos diferentes widgets que o dashboard possa apresentar e teve como resultados iniciais uma melhora significativa de performance, com a redução de tempo de carregamento a cada interação nos widgets.

2. Entendimento do problema

Foi definida com síntese do problema a seguinte frase: “As pessoas que utilizam os dashboards precisam navegar de forma fluida, rápida e assertiva pelos gráficos, porque atualmente elas têm dificuldade para explorar os dados no dashboard e em muitos casos o tempo de carregamento é muito longo".

3. Pesquisa

Foram feitas interações via slack e entrevistas semi-estruturadas com diferentes stakeholders para obtenção de dados qualitativos de comportamento, dores e expectativas.

Metodologia

  • Recrutamento: Stakeholders internos e externos de diferentes segmentos, maioria de O&G;
  • Participantes: 3 stakeholders externos e 9 stakeholders internos, dentre eles: engenheiros, cientistas de dados e SMEs (especialistas de domínio);
  • Formato: Entrevista semiestruturada (modelo flexível, com roteiro prévio, mas aberto para perguntas fora do planejado);

Coleta de dados

Após o processo de transcrição das entrevistas gravadas, foi feita a categorização das falas dos entrevistados em grandes grupos e, em seguida,os trechos categorizados foram divididos novamente, em subcategorias, a fim de facilitar a encontrabilidade dos dados por assunto e organizar as informações coletadas.

”E aí, por exemplo, depois que você dá zoom conseguir restaurar Né? O zoom movendo para frente, para trás em né? Dando mais agilidade na visualização dele no gráfico. Porque assim é você pensar na dinâmica do usuário, eu vou querer dar um zoom naquele momento para ver o que aconteceu, mas depois eu quero voltar para a visão macro, hã? Aí para voltar para a visão macro, eu vou ter que voltar ali no provavelmente ali no Calendário, e vou ter que falar que eu quero a última hora, último período.” (usuário entrevistado)

4. Proto Personas

Foram descobertos dois tipos de usuários, por tipo de consumo de dados.

Usuário que consome dados em tempo real

Conhecido de diferentes formas, dentre elas suporte operacional, engenheiro de avaliação de dados de performance ou specialist monitoring, é o usuário Interessado no que está acontecendo agora, passa a maior parte do tempo vendo dados em tempo real, mas às vezes volta um pouco no span de tempo pra ver recortes específicos e investigar pontos de interesse.

Usuário que consome dados históricos e em tempo real

Seja com cargo gerencial ou um engenheiro especialista, é um usuário que geralmente usa um gráfico temporal com span grande com canais que faça sentido olhar algum tipo de avanço ou mudança cuja relevância não ocorre "every second", mas sim "every hour", "every day", "every week", tanto dados históricos quanto em tempo real. Trata-se de um usuário mais interessado nos KPIs e na eficiência das operações e com a estabilidade do poço.

5. Desafio Sintetizado

Usando o framework “How might we?”, foram levantados diversos desafios, sintetizados na seguinte frase: “Como nós podemos permitir que a pessoa usuária interaja de forma mais prática e fluida com os gráficos ao navegar pelos dados, com um tempo de resposta reduzido?”

6. O Data Navigation

Diante do desafio sintetizado, foi desenvolvido o Data Navigation, composto por três features principais:

Feature 1: Zoom controls

Feature tem o objetivo de trazer melhorias de usabilidade na navegação manual do usuário dentro dos widgets do dashboard. A interação de zoom in e zoom out afeta todos os widgets que seguem o mesmo tipo de span.

Além da possibilidade de realizar zoom in e zoom out por meio do mouse wheel, o usuário pode fazer zoom in selecionando o trecho de interesse no gráfico e fazer zoom out dando duplo clique em qualquer local do widget.

Feature 2: Span control widget

Feature tem o objetivo de auxiliar as ferramentas de controle de zoom para navegação pelos dados, a nível de dashboard.

Além disso, é dada ao usuário, em sua maioria de O&G, a possibilidade de ver e controlar, no mesmo dashboard, trechos de interesse de um processo inteiro de perfuração.

Importante mencionar que se trata de um widget genérico, portanto a possibilidade de ver um span ”macro” pode ser expandida além do cenário de O&G.

Feature 3: Multiple Span Picker

Feature tem o objetivo de possibilitar que o usuário tenha controle de mais de um tipo de span no mesmo dashboard.

Além da possibilidade de ter mais de um tipo de span no mesmo dashboard, haverá também a possibilidade de vincular os dois tipos de span, de forma que o usuário selecione um intervalo no tempo e o sistema apresente, também, o intervalo de profundidade correspondente ao intervalo de tempo selecionado.

7. Resultados

Embora a INTELIE não possua estrutura para coleta de métricas de performance, foi observada uma melhoria significativa no tempo de carregamento a cada interação com as novas features de navegação, tendo em vista que anteriormente o sistema fazia uma nova requisição a cada interação do usuário com o widget, enquanto o Data Navigation tem como premissa deixar os dados pré carregados, sem realizar novas requisições a cada interação, resultando em interações de zoom in, por exemplo, sem tempo de espera de carregamento e renderização instantânea.

8. Conclusão

Ainda que o projeto tenha sido originado de uma hipótese já levantada como dor por alguns dos clientes listados neste documento, fazer um processo de pesquisa robusto foi muito importante não apenas para validar a hipótese levantada, mas também para levantar expectativas que estavam relacionadas aos problemas encontrados, que não estavam tão em evidência. Além disso, este foi o primeiro projeto que necessitou em seu processo o relacionamento entre diferentes equipes, como a equipe voltada para visualização de dados e a equipe voltada para performance da plataforma.

9. Desafios do Projeto

  • Realizar entrevistas com stakeholders externos e usuários;
  • Estudo aprofundado sobre domínio de O&G;
  • Alinhamento de priorização entre diferentes squads (nem todos as particularidades do projeto seriam desenvolvidas pela equipe na qual eu estava alocada);
  • Necessidade de alinhamentos constantes com lideranças;
  • Gerenciamento de mais de mais de uma feature gerada no mesmo produto;
